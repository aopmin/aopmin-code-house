## 1、概述

### 1.1 AI 基础概念

1. **LLM（Large Language Model）**：大语言模型，它是一种通过海量文本数据训练，能够理解和生成人类自然语言的人工智能系统。

2. **Token**：是大模型处理文本的基本单位，所有 LLM 服务商均按 Token 数量计费；通常情况下，1 个 Token 约等于 1 个汉字，或约 3-4 个字母。token 大小可通过厂商提供的可视化切词工具查看：
   - 文心一言：[https://console.bce.baidu.com/support/#/tokenizer](https://console.bce.baidu.com/support/#/tokenizer)
   - 阿里千问：[https://dashscope.console.aliyun.com/tokenizer](https://dashscope.console.aliyun.com/tokenizer)

3. **模型参数**：调用模型时可以设置一些参数，大部分 LLM 都支持的参数如下：

   - **Temperature（温度系数）**：控制生成文本的**随机性**。值越低(如 0.2)，输出越确定、保守和专注，值越高（如 0.9）输出越具有创造和多样性，但也可能更不连贯。
   - **Top-p（核采样）**：与 Temperature 类似，用于控制采样的多样性。它从累积概率超过阈值 p 的最可能候选词中随机选择。较低的值 (如 0.5) 限制选择范围，输出更可预测；较高的值 (如 0.9) 则扩大选择范围。
   - **Max Tokens（最大生成长度）**：限制模型单次响应所能生成的最大 token 数量。设置过短可能导致回答被截断。
   - **流式响应**：是指大模型返回结果时，并非一次性给出完整内容，而是像人打字一样，一个字一个字（或一段一段）逐步输出。目前流式响应主要是使用 SSE 技术实现。
   - **Message**：与大模型对话的数据载体，目前主要是 4 种消息角色：
     - system（系统消息，用于设定 AI 的行为和角色）
     - user（用户消息，来自用户的输入）
     - assistant（助手消息，来自 AI 的回复）
     - tool（工具调用消息）
   - **Prompt（提示词）**：传递给 AI 模型的指令或问题。简单来说，Prompt 就是需要让 AI 帮你做什么，以及怎么去做。

## 2、Spring AI Alibaba 介绍

## 4、SpringAI 核心 API

### 4.1 Message（消息）

![Message 结构](/code-house/images/agent/1765727979721-0942db0e-5719-4aa1-a3db-fd14a7e4da3e.png)

- 从上面请求体结构中可以看出，`messages` 它是一个数组
- Spring AI 会将 `messages` 数组中的每个对象抽象为 `Message`
- 而每个 message 对象，都包含 `role`（角色）和 `content`（内容）属性。

下面是豆包对 Message 的理解：

`org.springframework.ai.chat.messages.Message` 是 SA 与**大模型交互的核心数据载体**，所有对话内容（单轮 / 多轮、指令 / 请求 / 回复）都通过它封装传递。

Message 消息角色分为 4 类：（`org.springframework.ai.chat.messages.MessageType`）

- `SYSTEM`：系统消息（用于配置 AI 行为）；
- `USER`：用户消息（用户输入）；
- `ASSISTANT`：助手消息（AI 输出）；
- `TOOL`：工具消息（工具调用或工具返回结果）。

> 注：普通对话模型主要支持纯文本内容，不能包含系统消息或助手消息。

SpringAI 中 Message 接口的继承体系：

![Message 继承体系](/code-house/images/agent/1766420198351-8163791c-8bbb-4e27-84ec-ba4c5308fa0f.png)

`AbstractMessage` 是所有具体消息的抽象父类，继承自 `Content`，封装了通用属性：

- `textContent`：文本内容（可以理解是 content 的抽象）
- `messageType`：消息类型（可以理解是 role 的抽象，关联 `MessageType` 枚举）
- `metadata`：元数据

扩展：

```
1. SystemMessage
   消息类型：固定为 SYSTEM
   功能：传递系统级指令，用于设定 AI 的人设、规则等
2. UserMessage
   消息类型：固定为 USER
   功能：支持多模态输入
   关联对象：通过 media() 方法关联 Media 集合（如关联用户发送的图片等多媒体内容）
3. AssistantMessage
   消息类型：固定为 ASSISTANT
   功能：支持工具调用
   关联对象：通过 toolCalls 关联 ToolCall 集合
   ToolCall 属性：id 调用唯一标识，type 工具类型，name 工具名称，arguments 调用参数
4. ToolResponseMessage
   消息类型：固定为 TOOL
   功能：承载工具返回结果
   关联对象：通过 responses 关联 ToolResponse 集合
   ToolResponse（工具返回数据表示）属性：id（与 ToolCall 对应的标识），name 工具名称，responseData 工具返回的具体数据
```

> 那如何使用 Message？

上面的案例是通过 `chatModel.call()` 方法调用大模型，方法入参除了传普通文本字符串，还可以传 Message 可变参数。

![call 方法签名](/code-house/images/agent/1766422135187-0a6ff12a-2d1f-4d9c-8191-dbbd64cc4d2f.png)

示例代码：

```java
@GetMapping("/message")
public String chat(@RequestParam(name = "question") String question) {
    log.info("start! chat params: {}", question);
    // 配置系统侧 AI 身份
    SystemMessage systemMessage = new SystemMessage("你是一个情感专家，擅长倾听和安慰用户。");
    // 配置用户侧 提出的问题
    UserMessage userMessage = new UserMessage("请用中文回答我的问题：" + question);
    // 调用模型 call(Message... messages)
    String resp = zhiPuAiChatModel.call(systemMessage, userMessage);
    log.info("end! chat result: {}", resp);
    return resp;
}
```

### 4.2 Prompt（提示词）

Prompt 提示词的作用：引导大模型按照我们的结构化指令进行输出，而不是答非所问。

![Prompt 结构](/code-house/images/agent/1766426938312-cc1d6d59-c198-4ed5-873f-3f403cd886e1.png)

在 SpringAI 中，是把模型参数和消息列表的组合抽象为 Prompt。

`org.springframework.ai.chat.prompt.Prompt` 类的两个核心属性：

- `List<Message> messages`：存放 message 列表
- `ChatOptions chatOptions`：存放模型参数

Spring AI 将各类大模型的通用参数统一封装到 `ChatOptions` 类中，再由模型厂商分别提供一个自己专属的 `ChatOptions` 实现，扩展其独有的模型配置项。

示例：Prompt API 的使用

```java
@GetMapping("/prompt")
public ChatResponse chatV2(@RequestParam(name = "question") String question) throws JsonProcessingException {
    log.info("start! chatV2 params: {}", question);
    // message
    SystemMessage systemMessage = new SystemMessage("你是一个情感专家，擅长倾听和安慰用户。");
    UserMessage userMessage = new UserMessage("请用中文回答我的问题：" + question);
    // chatoptions
    ZhiPuAiChatOptions zhiPuAiChatOptions = new ZhiPuAiChatOptions();
    zhiPuAiChatOptions.setModel("glm-4.5"); // 设置模型
    zhiPuAiChatOptions.setTemperature(0.0); // 设置温度
    zhiPuAiChatOptions.setMaxTokens(15536); // 设置最大生成长度
    // 组装提示词 prompt
    Prompt prompt = new Prompt(List.of(systemMessage, userMessage), zhiPuAiChatOptions);
    // 调用模型接口
    return zhiPuAiChatModel.call(prompt);
}
```

### 4.3 ChatModel（对话模型）

![ChatModel](/code-house/images/agent/1766502652619-3cb32327-e116-4663-8527-33c0dda56cd2.png)

ChatModel 是 Spring AI 框架中对话交互的底层核心接口，不仅定义了 `call()` 同步调用，还提供了 `stream()` 流式调用，让应用开发者非常方便地与 AI 大模型进行文本交互。

同时，它对应用与模型层的交互进行了统一抽象，包括使用 Prompt 作为输入，使用 ChatResponse 作为输出等。

### 4.4 ChatClient（对话客户端）

`ChatModel` 是实现 AI 模型调用的底层核心接口，但它需要开发者手动完成输入组装、输出解析等一系列操作，使用起来比较繁琐。

而 Spring AI 提供的 `ChatClient` 封装类，不仅能大幅简化模型的使用流程，还内置了许多 `ChatModel` 不具备的额外功能，提升开发效率。

ChatClient 基础功能：

- 定制和组装模型的输入（Prompt）
- 格式化解析模型的输出（Structured Output）
- 调整模型交互参数（ChatOptions）
- ...

ChatClient 高级功能：

- 聊天记忆(Chat Memory)：维持多轮对话上下文
- 工具/函数调用(Function Calling)：模型调用外部工具
- RAG：检索增强生成，结合外部知识库提升回答质量

#### 创建 ChatClient

```java
@Slf4j
@Configuration
public class SaaLLMConfig {

    /**
     * chatClient 不支持自动注入 bean，因此需要手动注入
     * @param zhiPuAiChatModel   chatModel
     * @return  zhiPuAiChatClient chatClient
     */
    @Bean
    public ChatClient zhiPuAiChatClient(ChatModel zhiPuAiChatModel) {
        ChatClient chatClient = ChatClient.builder(zhiPuAiChatModel).build();
        log.info("===dashScopeChatClient bean 创建成功===");
        return chatClient;
    }
}
```

#### 模型调用并处理响应

**（1）返回字符串**

```java
package cn.aopmin.controller;

import lombok.extern.slf4j.Slf4j;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.zhipuai.ZhiPuAiChatOptions;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@Slf4j
@RestController
@RequestMapping("/chatclient")
public class ChatClientController {

    @Autowired
    private ChatClient chatClient;

    @RequestMapping("/simple")
    public String simple(String query) {
        log.info("start! simple. query:{}", query);
        ZhiPuAiChatOptions zhiPuAiChatOptions = ZhiPuAiChatOptions.builder()
                .maxTokens(15536)
                .temperature(0.0)
                .model("glm-4.5")
                .build();
        String res = chatClient.prompt()
                .system("你是一个有用的AI助手。")
                .user("请用中文回答我的问题：" + query)
                .options(zhiPuAiChatOptions)
                .call().content();
        log.info("end simple. res:{}", res);
        return res;
    }
}
```

**（2）返回 ChatResponse 数据**

```java
/**
 * 返回ChatResponse
```

**（3）返回值以实体类返回**

```java
/**
 * 响应数据转成实体
 */
@RequestMapping("/simple/entity/resp")
public Book simpleEntityResp(@RequestParam("query") String query) {
    log.info("start! simpleResp. query:{}", query);
    ZhiPuAiChatOptions zhiPuAiChatOptions = ZhiPuAiChatOptions.builder()
            .maxTokens(15536)
            .temperature(0.0)
            .model("glm-4.5")
            .build();
    // 用chatClient调用完call方法后调用entity方法传入要转化类的字节码对象
    return chatClient.prompt()
            .system("你是一个有用的AI图书助手。")
            .user("请用中文回答我的问题：" + query)
            .options(zhiPuAiChatOptions)
            .call().entity(Book.class);
}

@Data
class Book {
```

### 4.5 Advisors

#### 4.5.1 概述

> 文档：[https://springdoc.cn/spring-ai/api/advisors.html](https://springdoc.cn/spring-ai/api/advisors.html)

Spring AI Advisors 是 Spring AI 框架中的**核心拦截器组件**，专门用于处理和增强 AI 应用程序中的请求与响应流。其设计与过滤器、拦截器非常类似。

**Advisors 的优势：**

- 将对话记忆、敏感词过滤、RAG 检索等高频 AI 场景，打包为可直接复用的组件，简化开发流程、提升交付效率。
- 助力打造可跨不同 AI 模型、不同业务用例复用的数据转换组件，实现"一次开发，多场景适配"；切换 AI 模型仅需修改配置，需求变更仅需调整转换组件，显著降低系统维护与扩展成本。

以下流程图展示了 Advisor 链与聊天模型的交互过程：

![Advisor 流程](/code-house/images/agent/1766624436637-e9df42e3-0681-46e6-951c-3582acbe3ea7.png)

交互过程：

```
1、Spring AI 框架将用户的 Prompt 封装为 AdvisedRequest 对象，并创建空的 AdvisorContext 上下文。
2、Advisor 链中每个 Advisor 依次处理请求并可进行修改，也可选择阻断请求（不调用下一实体）。若选择阻断，该 Advisor 需负责填充响应内容。
3、框架提供的最终 Advisor 将请求发送给 ChatModel，然后进行模型调用。
4、ChatModel 的响应会逆向传回 Advisor 链，被转换为包含共享 AdvisorContext 实例的 AdvisedResponse 对象。
5、每个 Advisor 均可处理或修改该响应。
6、通过提取 ChatCompletion 内容，最终生成的 AdvisedResponse 数据响应给客户端。

总结：Advisor 可以理解为 AOP 切面，在与 AI 交互时对请求和响应进行增强处理。
```

继承体系：

![Advisor 继承体系](/code-house/images/agent/1766625318276-d8d7d96c-7fdc-4c13-b77f-b488d3d6a795.png)

- 如果我们需要对同步调用进行增强可以使用 `CallAdvisor`
- 如果是对流式调用(响应式调用)进行增强可以使用 `StreamAdvisor`

> MessageChatMemoryAdvisor 源码分析：

```
1、MessageChatMemoryAdvisor：底层是AOP环绕通知，只做2件事
    - before 前置增强：提取请求中用户Message → 存入 ChatMemory → 拼装上下文请求，调用大模型
    - after 后置增强：提取响应中AI回复Message → 存入 ChatMemory → 返回最终响应

2、ChatMemory 是一个顶级接口，用来存储和管理对话记录的。内部只定义3个标准方法：
   add(会话ID,消息)、get(会话ID)、clear(会话ID)
   官方提供的默认实现是 MessageWindowChatMemory。

3、MessageWindowChatMemory 内部核心属性：
    - DEFAULT_MAX_MESSAGES = 20：默认最多存储 20 条对话消息（用户 + AI 消息算 2 条）
    - chatMemoryRepository：底层真实存储载体
    - maxMessages：消息窗口阈值，超过自动裁剪

4、chatMemoryRepository 是一个接口，官方默认实现是 InMemoryChatMemoryRepository（内存存储），
   底层把会话消息放到 map 集合中，Key 为会话ID，Value 为有序消息列表
    - 存储容器：private final Map<String, List<Message>> memoryStore = new ConcurrentHashMap<>()
    - 存储介质：JVM 堆内存，纯内存数据，服务重启后数据全丢失
    - 线程安全：ConcurrentHashMap 天然支持多线程并发对话

5、扩展规则：如需存储至 Redis / 数据库 / 文件，仅做接口扩展即可，业务代码无任何修改
    - 推荐首选：扩展 ChatMemoryRepository 接口（底层存储接口，只实现消息 CRUD）
    - 兜底方案：扩展 ChatMemory 接口（顶层管理接口，完全自定义逻辑）
```

## 5、RAG

### 5.1 概述

> 智普文本嵌入接口文档：[https://docs.bigmodel.cn/api-reference/模型-api/文本嵌入](https://docs.bigmodel.cn/api-reference/%E6%A8%A1%E5%9E%8B-api/%E6%96%87%E6%9C%AC%E5%B5%8C%E5%85%A5#body-dimensions)

如果用户的问题超出了大模型的预训练知识范围，AI 就很容易凭空捏造内容，也就是我们常说的幻觉问题（答非所问）。

如果我们想要避免 AI 出现幻觉，就需要让 AI 知道相关知识。

RAG（检索增强生成）核心价值在于，构建一个知识库（外部数据库），通过知识库检索知识，然后把问题和相关知识一起给大模型，让大模型生成回答。

![RAG 流程图](/code-house/images/agent/1767108981942-460c47e9-da54-4d52-86a5-18461bf6cfa7.png)

相关概念：

- **向量**：向量就像是一个有序的数字列表，或者说是多维空间中的一个点、一条有方向的线段。这些数字共同描述了对象在某一个"特征空间"中的位置。
- **为什么要把文本转换成向量**：计算机无法直接理解人类语言，但极其擅长处理数字。一旦文本变成了向量（即高维空间中的点），我们就可以运用强大的数学工具来处理它们，计算相似度（比如余弦相似度算法）。
- **Embedding（嵌入）**：把文本 -> 多维向量的过程叫做 Embedding 嵌入。
- **EmbeddingModel（嵌入模型）**：嵌入模型可以把文本转换成多维向量。
- **向量数据库**：能够存储向量并且具有向量相似度检索相关能力的数据库。
- **向量维度**：向量的维度就是这个列表里数字的个数。较高维度通常能提供更丰富的语义信息和更强的区分能力，但这会增加计算量，占用更多内存和存储。

官网文本转向量的示例：

```bash
curl --request POST \
  --url https://open.bigmodel.cn/api/paas/v4/embeddings \
  --header 'Authorization: Bearer <token>' \
  --header 'Content-Type: application/json' \
  --data '
{
  "model": "embedding-3",
  "input": "你好，今天天气怎么样.",
  "dimensions": 2
}
'
```

```java
package cn.aopmin.controller;

import lombok.extern.slf4j.Slf4j;
import org.springframework.ai.document.Document;
import org.springframework.ai.vectorstore.SearchRequest;
import org.springframework.ai.vectorstore.VectorStore;
import org.springframework.util.Assert;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

import java.util.List;

@Slf4j
@RestController
@RequestMapping("/redis")
public class RedisController {

    private final VectorStore vectorStore;

    public RedisController(VectorStore vectorStore) {
        this.vectorStore = vectorStore;
    }

    /**
     * 向量数据存储
     */
    @RequestMapping("/save")
    public Result save(@RequestParam("data") String data) {
        log.info("begin save vector data to redis. param: {}", data);
        try {
            Assert.hasText(data, "入参不能为空");
            List<Document> documents = List.of(new Document(data));
            vectorStore.add(documents);
            log.info("save vector data to redis success!");
        } catch (Exception e) {
            log.error("save vector data to redis error!", e);
            return Result.fail(500, "保存数据失败");
        }
        return Result.success();
    }

    /**
     * 相似度搜索（语义搜索）
     */
    @RequestMapping("/search")
    public Result search(@RequestParam("query") String query) {
        log.info("begin search vector data from redis. param: {}", query);
        try {
            SearchRequest searchRequest = SearchRequest.builder()
                    .topK(2)
                    .query(query)
                    .similarityThreshold(0.8)
                    .build();
            List<Document> documents = vectorStore.similaritySearch(searchRequest);
            log.info("search vector data from redis success! result: {}", documents);
            return Result.success(documents);
        } catch (Exception e) {
            log.error("search vector data from redis error!", e);
            return Result.fail(500, "搜索数据失败");
        }
    }
}
```

```java
                    .setHeader()
                    .setSkipHeaderRecord(true)
                    .build().parse(reader);
            List<Document> documents = new ArrayList<>();
            for (CSVRecord record : csvParser) {
                String question = record.get("问题");
                String answer = record.get("回答");
                String content = "问题: " + question + "\n回答: " + answer;
                Document document = new Document(content);
                documents.add(document);
            }
            csvParser.close();
            vectorStore.add(documents);
            return "成功导入 " + documents.size() + " 条记录到向量数据库";
        } catch (IOException e) {
            e.printStackTrace();
            return "导入失败: " + e.getMessage();
        }
    }
}
```

```java
    }
```

## 6、ToolCalling（工具调用）

### 6.1 ToolCalling 解决了什么问题

如果问智能客服说：我现在下单一杯美式几点几分可以做好

你会发现他没办法很好的回答问题。因为模型能力是有限的（比如不知道当前时间），如果想要让 AI 应用具备更强的功能，就可以使用 Tool Calling。

相当于是给模型提供一些工具，当模型判断需要使用工具的时候就会进行使用。

### 6.2 ToolCalling 调用流程

![ToolCalling 流程](/code-house/images/agent/1767205667846-38beeedb-7058-4540-a7c5-d740764e21fb.png)

其实就是把工具的 `name`、`description`、`input schema` 等信息也放入模型的请求中。

模型如果觉得需要调用工具就会返回相关的工具调用消息，这个时候 SpringAI 会基于这个消息去调用工具，得到一个工具调用的结果给到模型。

模型再综合这个调用结果返回最终的答案。

### 6.3 ToolCalling 使用

1. 定义工具

```java
package cn.aopmin.tool;

import org.springframework.ai.tool.annotation.Tool;
import org.springframework.ai.tool.annotation.ToolParam;

import java.time.ZoneId;
import java.time.ZonedDateTime;
import java.time.format.DateTimeFormatter;

public class TimeTools {
    @Tool(description = "通过时区id获取当前时间")
    public String getTimeByZoneId(@ToolParam(description = "时区id, 比如 Asia/Shanghai") String zoneId) {
        ZoneId zid = ZoneId.of(zoneId);
        ZonedDateTime zonedDateTime = ZonedDateTime.now(zid);
        DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss z");
        return zonedDateTime.format(formatter);
    }
}
```

2. 传入工具

## 7、MCP

> [https://www.mcpworld.com/](https://www.mcpworld.com/)

MCP（Model Context Protocol）模型上下文协议，是一种开放标准，它规范了应用程序如何向大语言模型提供上下文信息，从而实现大模型与外部工具、数据源的标准化连接。

简单来说它是解决大模型与外部世界交互的问题，让 AI 模型能够无缝调用各种工具、访问数据源并执行复杂操作。

![MCP 架构](/code-house/images/agent/1767232511960-89afac81-e738-479c-8af4-81c145db6291.png)

MCP 相关概念：

- **MCP Host（MCP 主机）**：使用 MCP 协议调用工具的应用程序。例如 Claude Code、我们自己写的 Spring AI 应用等。它负责协调和管理一个或多个 MCP 客户端的 AI 应用程序。
- **MCP Server（MCP 服务器）**：给 AI 模型提供工具的服务。
- **MCP Client（MCP 客户端）**：与 MCP 服务器保持连接并从 MCP 服务器获取上下文，供 MCP 主机使用的组件。

![MCP 交互](/code-house/images/agent/1767232936170-2bc55e49-0b52-4df8-9204-8a7cf52c2490.png)

## 8、Graph

### 8.1 为什么需要 Graph

假设你需要处理一个真实的商业需求：比如自动化审核一份商业合同。

拆分：

```
- 理解内容：先请AI通读合同，提炼摘要和关键条款。
- 合规检查：再让AI根据公司政策库，判断合同是否有合规风险。
- 风险评估：接着，要求AI从法律、财务等多维度给出风险评分。
- 人工介入：最后，必须将AI的分析结果提交给法务专家，由专家做出"批准"、"拒绝"或"要求修改"的最终决定。
- 后续动作：系统根据专家的决策，自动进入不同的处理流程（如生成公文、起草拒绝函等）。
```

你会发现它变成了一个多步骤、有状态、且需要协调 AI 自动化和人类决策的复杂流程。这就是我们常说的 **AI 工作流（AI Workflow）** 或**智能体（AI Agent）** 要处理的核心问题。

在没有 Graph 之前，实现这样的流程会非常麻烦，需要我们写很多硬编码去编排实现。

而 Spring AI Alibaba Graph 就是为了优雅地解决这些问题而生的。它是一个强大的工作流编排引擎，让你能像画流程图一样，直观地定义和执行复杂的 AI 应用流程。

### 8.2 Graph 核心概念

- **State（状态）**：负责在不同步骤间安全地传递和共享数据的容器。
- **Node（节点）**：代表一个执行单元，可以是一个 AI 调用、一个数据库操作、或一段业务逻辑。
- **Edge（边）**：定义节点之间的连接关系和流转方向。可以是固定的简单边，也可以是能根据 State 内容决定下一步的条件边。

5. 编写代码：定义状态图（状态，节点，边），编译生成 CompiledGraph 后注入 Spring 容器

```java
package cn.aopmin.config;

import com.alibaba.cloud.ai.graph.*;
import com.alibaba.cloud.ai.graph.action.NodeAction;
import com.alibaba.cloud.ai.graph.exception.GraphStateException;
import com.alibaba.cloud.ai.graph.state.strategy.ReplaceStrategy;
import lombok.extern.slf4j.Slf4j;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import java.util.Map;
import static com.alibaba.cloud.ai.graph.action.AsyncNodeAction.node_async;

@Slf4j
@Configuration
public class GraphConfiguration {

    @Bean("quickStartGraph")
    public CompiledGraph quickStartGraph() throws GraphStateException {
        // 1、使用 StateGraph 定义状态图
        KeyStrategyFactory keyStrategyFactory = new KeyStrategyFactory() {
            @Override
            public Map<String, KeyStrategy> apply() {
                return Map.of(
                        "input1", new ReplaceStrategy(),
                        "input2", new ReplaceStrategy()
                );
            }
        };
        StateGraph stateGraph = new StateGraph("quickStartGraph", keyStrategyFactory);

        // 2、定义节点1
        stateGraph.addNode("node1", node_async(new NodeAction() {
            @Override
            public Map<String, Object> apply(OverAllState state) throws Exception {
                log.info("node1 state: {}", state);
                return Map.of("input1", 1, "input2", 1);
            }
        }));
        // 定义节点2
        stateGraph.addNode("node2", node_async(new NodeAction() {
            @Override
            public Map<String, Object> apply(OverAllState state) throws Exception {
                log.info("node2 state: {}", state);
                return Map.of("input1", 2, "input2", 2);
            }
        }));

        // 3、定义边 开始节点 -> 节点1 -> 节点2 -> 结束节点
        stateGraph.addEdge(StateGraph.START, "node1");
        stateGraph.addEdge("node1", "node2");
        stateGraph.addEdge("node2", StateGraph.END);

        // 4、编译状态图
        return stateGraph.compile();
    }
}
```
